# 开发计划 (Roadmap)

基于系统作为“AI 友好的个人数据底座”这一核心定位，开发计划将分为以下四个阶段（Phases）进行迭代推进。

---

## Phase 1: 核心中台与 API (后端优先)

**目标：** 建立精简后的数据库模型，实现核心的事件/草稿 CRUD、API Key 鉴权以及 URL 基础解析工具，为前端和外部 Agent 提供坚固的数据基础。

### 后端 (Server)
- [x] 项目脚手架与基础框架搭建（Fastify + TypeScript + 目录结构设计）
- [x] 数据库连接与初始化脚本（基于 `sqlite3` 创建简化的表结构）
- [x] 基础 API 开发：
  - [x] 事件 CRUD API (包含 `status` 状态字段支持)
  - [x] 标签体系 CRUD API（支持插入未存在的标签即时创建）
- [x] 系统与安全审计：
  - [x] `api_keys` 管理接口
  - [x] API 请求鉴权中间件 (验证 `X-API-Key`)
  - [x] 简易 `operation_logs` 拦截写入逻辑
- [x] **增强工具**：
  - [x] URL 解析服务 (`/api/tools/parse-url`): 利用 Readability 提取标题和 Markdown 正文

### 前端 (Web)
- [x] 前端项目初始化（Vue 3 + Vite + Naive UI）
- [x] 基础网络请求封装（Axios + 统一错误处理拦截）
- [x] 路由搭建与空页面占位

---

## Phase 2: 核心事件库与收件箱 (前端优先)

**目标：** 确立“手动录入与正式事件库”的核心地位，并打通外部 Agent 投递至收件箱的审批闭环。

### 前端 (Web)
- [x] **正式事件库视图 (Events List - 优先级最高)**：
  - [x] 展示已确认 (`confirmed`) 的时事列表及分页
  - [x] 提供强大的“手动添加事件”按钮与弹窗（默认状态即为 `confirmed`）
  - [x] 弹窗内集成“URL 智能提取”功能，填充表单供用户确认后保存
  - [x] 标签选择器支持即时输入即时创建
- [x] **收件箱 (Inbox) 看板开发**：
  - [x] 拉取并展示 `status='draft'` 的待处理事件列表
  - [x] 提供一键"收录 (Confirm)" 与 一键"忽略 (Archive)"操作
- [x] **系统设置视图**：
  - [x] 提供管理、生成 API Key 的 UI 界面

### 后端 (Server)
- [x] 配合前端联调，修复 API 格式与标签即时创建逻辑
- [x] **严格权限控制**：实现当事件状态为 `confirmed` 时，拒绝 Agent（API Key 来源）修改事件主标题和正文的拦截逻辑

---

## Phase 3: 标签聚类与全文检索

**目标：** 实现完善的标签分类展示、事件详情查阅与基于 SQLite FTS5 的全局搜索功能。

### 后端 (Server)
- [x] 部署 SQLite FTS5 虚拟表及数据同步触发器
- [x] 编写 `/api/search` 全局搜索接口（支持 FTS5 全文搜索和日期范围过滤）

### 前端 (Web)
- [x] **事件详情页开发**：
  - [x] 详情展示与 Markdown 渲染
  - [x] 支持编辑事件正文、修改关联标签
- [x] **标签管理侧边栏/视图**：
  - [x] 树形标签展示与快捷筛选
- [x] **全局搜索交互**：
  - [x] 顶部全局搜索框与结果高亮

---

## Phase 4: 深度资料管理与时间线

**目标：** 提供深度的事件脉络追踪能力，允许挂载多媒体参考材料（Materials）与时间线节点（Timeline Nodes），并赋予用户完全的手动编辑权限。

### 后端 (Server)
- [x] 时间线节点 CRUD API (`/api/events/:id/timeline`)
- [x] **文件系统与材料 API**：
  - [x] 实现安全的文件上传机制与本地目录归档 (`/api/materials/upload`)
  - [x] 材料查询、删除与预览服务

### 前端 (Web)
- [x] **事件详情页增强**：
  - [x] 内嵌时间线组件，支持用户手动**增加、修改、删除**发展脉络
  - [x] 内嵌参考材料附件列表，支持用户**上传、修改、删除**图片或文档

---

## Phase 5: 远期规划 (Future)

**目标：** 随着系统成熟和场景扩大，逐步引入更为复杂的安全与协作机制。
- [ ] **多用户协作与登录认证**：在 Web 端引入账号体系（如基础的 JWT 登录或 OAuth 接入），以支持部署在公网环境下的安全性与多成员共享记录能力。
- [ ] **数据清理策略**：根据真实使用情况的存储压力，设计数据的物理删除（Hard Delete）或冷归档策略。

---
