# 岁月史书 (Book of Ages) - 核心设计文档

**文档日期**: 2026-04-02
**项目目标**: 构建一个作为“纯净数据底座”的个人热点时事记录工具，通过提供对 AI 友好的 API，配合外部 Agent（如 OpenClaw 定时脚本）实现自动化信息采集、提炼，并在内部 UI 提供极简高效的“收件箱审批与收录”体验。

## 1. 系统架构与核心流程

本系统（Book of Ages）自身**不包含任何大语言模型（LLM）的直接调用与主动分析逻辑**。它是一个高内聚的 Headless CMS（无头内容管理系统）与个人 UI 展示端的结合体。

### 核心流转机制 (Inbox/Approval Flow)
1. **自动搜寻（外部）**：外部爬虫/时事 Agent 定时运行，搜寻并总结当天的热点事件。
2. **批量推送（API）**：Agent 通过带有鉴权（API Key）的 RESTful API，将抓取和总结的事件以 `status='draft'` 的状态推送到本系统。
3. **人工审批（前端 UI）**：用户在 Web 首页的“待处理 / 收件箱 (Inbox)”中浏览 Agent 抓取的事件梗概。
4. **一键收录/审批**：用户点击“确认收录 (Confirm)”，事件状态变更为 `confirmed`，正式进入史书档案；点击“忽略/归档”，则变更为 `archived` 或软删除。
5. **深度资料补全（可选）**：在人工确认收录后，外部 Agent 可通过 API 继续向该事件追加细化的时间线节点（Timeline Nodes）或详细的参考材料（Materials）。

## 2. 核心功能设计

### 2.1 AI 友好的 API 接口与鉴权底座
- **严格鉴权**：系统保留 `api_keys` 表，为不同的外部 Agent 分配特定的 Token（如：爬虫专用 Token、资料收集专用 Token）。
- **高容错与规范响应**：所有的 API（特别是供 Agent 调用的 `/api/events` 和 `/api/materials`）必须提供极度规范的 JSON 输入输出格式，并带有明确的错误代码，以便外部脚本重试纠错。
- **基础防灾审计**：保留简化的 `operation_logs` 表，仅记录审计级别的动作（“哪个 API Key 在什么时间对什么实体进行了何种 CRUD 操作”），不做详细的字段级 Diff 快照对比，用于快速追踪异常 Agent 行为。

### 2.2 核心录入体验优化（一键剪藏提效）
- **URL 解析服务 (`/api/tools/parse-url`)**：
  - **功能**：接收一个网页 URL，后端自动利用 Readability 算法或类似工具提取网页的“标题”、“正文（Markdown 格式输出）”，并可选地保存网页基础截图（Snapshot）。
  - **用途**：无论是外部 Agent 传入原始链接进行归档，还是用户手动在 UI 粘贴新闻链接，该接口都能极大地免除手动复制正文与排版的工作量。

### 2.3 极简数据模型 (YAGNI 原则调整)
为了保证系统能快速交付，对原有的复杂数据模型进行以下精简：
- **移除复杂的版本控制**：删除 `event_versions` 表。系统有明确的“草稿-审批”流且为单人使用，前期无需细粒度的历史记录回滚。
- **简化事件关联**：删除原有带“AI 置信度”、“关联原因”等复杂字段的 `event_relations` 表。前期主要依靠“标签体系 (Tags)”来对事件进行聚类，或仅使用一个简单的 ID 数组来表达“相关链接”。

## 3. 技术栈保持不变

| 层级 | 技术选型 | 说明 |
|------|------|------|
| **前端 UI** | Vue 3 + Vite + Naive UI | 提供直观的收件箱看板、事件列表、标签管理与详情展示 |
| **后端 API** | Fastify (TypeScript) | 提供极速、规范的 RESTful API |
| **数据存储** | SQLite + 文件系统 | 基于 `better-sqlite3`，本地单文件存储，易于备份与 NAS 部署 |
| **全文检索** | SQLite FTS5 | 用于对事件标题、Markdown 正文与摘要进行极速搜索 |

## 4. 后续开发计划建议（Phase 调整）

- **Phase 1: 核心中台与 API (后端优先)**：建立精简后的数据库模型，实现核心的事件/草稿 CRUD、API Key 鉴权以及 URL 基础解析工具。
- **Phase 2: 收件箱与审批流 (前端优先)**：构建 UI “待处理看板”，打通“外部 Agent 推送 -> 人工浏览 -> 确认收录”的完整闭环链路。
- **Phase 3: 标签聚类与全文检索**：实现完善的标签分类展示、事件详情查阅与基于 SQLite FTS5 的全局搜索功能。
- **Phase 4: 深度资料管理与时间线**：允许外部 Agent 针对已确认的事件，持续挂载多媒体参考材料（Materials）与时间线节点（Timeline Nodes）。
